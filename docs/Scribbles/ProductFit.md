# ProductFit
---

```
Imagine investing time, effort, and resources into creating a product, only to find that no one wants it.  
  
It’s no fun.  
  
When approaching innovation as engineers, we often dive headfirst into the technicalities, the 'how-to's, and start building.  
  
We're problem solvers by nature, and the challenge of making something possible is what drives us. However, in this process, a vital question often takes the back seat - "Do people actually want or need what we're creating?"  
  
We want to build it, so surely someone will find it useful and pay for it. Right?  
  
Instead of asking "Can we build this?" - we should start by asking "Should we build this"?  
  
This shift is more than just a process change; it's about creating value and relevance.  
  
With limited resources, we cannot afford to create solutions in search of problems.

- Marcus
```

```
 very much agree with this - very clear!  
  
To add to this:  
I like to go through these filtering steps for an idea.  
  
1: Ask: "What is the problem we are solving?". The question engineers tend to ask is "What is it I want the tool to do?"  
  
2: Manually create a sample of an input and a desired output. This will help answer the question 'what does the tool need and what does it spit out'.... and it helps identify whether the answer to the first question is really valid.  
  
3: Place it in one of four categories in order of growing complexity and challenge of implementation:  
a: Existing process we wish to optimize on a single project (these have high succes rate).  
b: Existing process we wish to optimize on MANY projects. (Start at 'a' to validate your assumption)  
c: Something new we want to do on a single project (for example generative optimization or something else we don't normally do. This will always be extra work, so the value it brings has to be accordingly great!)  
d: Something new we want to do on MANY projects (start with 'c' to validate idea)

- Timo
```
